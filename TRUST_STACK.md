## Data Flow Architecture

### How Verification Works (End-to-End)

```
┌──────────────┐
│   USER       │ 1. Makes claim: "I'm a qualified builder"
│  (Builder)   │
└──────┬───────┘
       │
       │ 2. System gathers evidence (privacy-preserving)
       ↓
┌─────────────────────────────────────────────────────────┐
│  BELIEF SCORING ENGINE                                  │
│  ├─ GitHub Activity (Commits, PRs, Reviews)            │
│  ├─ On-Chain Behavior (Deploys, Interactions)          │
│  ├─ Community Signals (Attestations, Endorsements)     │
│  └─ External APIs (Optional: NPM, PyPI, etc.)          │
└─────────────┬───────────────────────────────────────────┘
              │
              │ 3. Multi-source score calculated
              ↓
┌─────────────────────────────────────────────────────────┐
│  ZERO-KNOWLEDGE PROOF GENERATION                        │
│  ├─ Score: 8,542 / 10,000 (85.42% = "High Believer")  │
│  ├─ Proof Generated: "Score ≥ 80%" ✅                   │
│  └─ Privacy: Score value HIDDEN from verifier          │
└─────────────┬───────────────────────────────────────────┘
              │
              │ 4. Proof submitted to blockchain
              ↓
┌─────────────────────────────────────────────────────────┐
│  SMART CONTRACT VERIFICATION                            │
│  ├─ Verifies: Proof is cryptographically valid         │
│  ├─ Verifies: Proof is recent (not stale/replayed)     │
│  ├─ Verifies: Threshold met (≥80%)                     │
│  └─ Records: ✅ Verified (NO personal data stored)      │
└─────────────┬───────────────────────────────────────────┘
              │
              │ 5. Access granted
              ↓
┌──────────────┐
│   PARTNER    │ Learns: "This builder is verified ✅"
│  (Platform)  │ Does NOT learn: Name, exact score, activity details
└──────────────┘
```

**Key Innovation:** Partner gets verification WITHOUT surveillance.

## Security Architecture

### Multi-Layer Defense System

```
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 1: Access Control                                       │
├─────────────────────────────────────────────────────────────────┤
│  ✅ Owner-only functions (pause, upgrade, emergency)           │
│  ✅ Reviewer-only functions (oracle validation, dispute)       │
│  ✅ Role-based permissions (modifiers on all functions)        │
│  ✅ Two-step ownership transfer (prevent accidents)            │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 2: Financial Safety                                     │
├─────────────────────────────────────────────────────────────────┤
│  ✅ ReentrancyGuard on ALL distribution functions              │
│  ✅ 7-day timelock before distributions                        │
│  ✅ Pull over push pattern (users withdraw, not auto-send)     │
│  ✅ Safe math (Solidity 0.8.20 built-in overflow protection)   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 3: Input Validation                                     │
├─────────────────────────────────────────────────────────────────┤
│  ✅ Score validation (0-10,000 range enforced)                 │
│  ✅ Address validation (no zero addresses)                     │
│  ✅ Amount validation (no zero amounts)                        │
│  ✅ Array bounds checking (prevent out-of-bounds access)       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 4: Oracle Security                                      │
├─────────────────────────────────────────────────────────────────┤
│  ✅ Multi-oracle consensus (no single point of truth)          │
│  ✅ Weighted median (Byzantine fault tolerant)                 │
│  ✅ Oracle staking (economic incentive for honesty)            │
│  ✅ Reputation scoring (bad oracles lose weight)               │
│  ✅ Dispute mechanism (challenge false data)                   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  LAYER 5: Emergency Controls                                   │
├─────────────────────────────────────────────────────────────────┤
│  ✅ Pausable (emergency stop for all bonds)                    │
│  ✅ Bug bounty (incentivize white-hat disclosure)              │
│  ✅ Rate limiting (prevent spam/DOS)                           │
│  ✅ No upgradeable proxies (fixes ship as new deployments +    │
│     migrations; existing contracts are immutable once live)    │
└─────────────────────────────────────────────────────────────────┘
```

## Economic Flow Architecture

### How Money Moves (BuilderBeliefBonds Example)

```
STEP 1: BOND CREATION
┌──────────────┐
│   BUILDER    │ Stakes 1 ETH
└──────┬───────┘
       │
       ↓ 1 ETH deposited
┌─────────────────────────┐
│  BuilderBeliefBonds.sol │
│  └─ bonds[1].stake = 1 ETH
└─────────────────────────┘

STEP 2: VALUE APPRECIATION (Over Time)
┌─────────────────────────────────────────────────┐
│  Bond Value Calculation (Every Block)          │
├─────────────────────────────────────────────────┤
│  buildingScore    = 8,500 / 10,000 = 85%      │
│  vestingMultiplier = 120 / 100 = 1.2x          │
│  timeMultiplier    = 180 / 100 = 1.8x          │
│                                                 │
│  bondValue = (1 ETH × 8,500 × 120 × 180)       │
│              ─────────────────────────          │
│                    50,000,000                   │
│            = 3.67 ETH                           │
│                                                 │
│  appreciation = 3.67 - 1.0 = 2.67 ETH          │
└─────────────────────────────────────────────────┘

STEP 3: DISTRIBUTION REQUEST
┌──────────────┐
│   BUILDER    │ Requests distribution
└──────┬───────┘
       │
       ↓ Timelock starts (7 days)
┌─────────────────────────────────────────────────┐
│  BuilderBeliefBonds.sol                         │
│  └─ bonds[1].distributionPending = true         │
│     bonds[1].availableAt = now + 7 days         │
└─────────────────────────────────────────────────┘

STEP 4: DISTRIBUTION (After 7-Day Timelock)
┌─────────────────────────────────────────────────┐
│  Distribution Calculation                       │
├─────────────────────────────────────────────────┤
│  appreciation = 2.67 ETH                        │
│                                                 │
│  builderShare  = 2.67 × 60% = 1.60 ETH   ──┐   │
│  stakerShare   = 2.67 × 30% = 0.80 ETH     │   │
│  ecosystemFund = 2.67 × 10% = 0.27 ETH     │   │
│                                             │   │
│  Total: 1.60 + 0.80 + 0.27 = 2.67 ✅       │   │
└─────────────────────────────────────────────┼───┘
                                              │
       ┌──────────────────────────────────────┘
       │
       ↓ 1.60 ETH
┌──────────────┐
│   BUILDER    │ Receives 60% (earned through building)
└──────────────┘

       ↓ 0.80 ETH
┌──────────────┐
│   STAKERS    │ Receive 30% (early believers rewarded)
└──────────────┘

       ↓ 0.27 ETH
┌──────────────┐
│  ECOSYSTEM   │ Receives 10% (fund future builders)
└──────────────┘
```

**Key Feature:** Distribution math ALWAYS sums to 100% (verified in code).

## Scalability Architecture

### How Vaultfire Scales to Billions

```
┌─────────────────────────────────────────────────────────────┐
│  LAYER 1: Blockchain (Base Mainnet)                         │
├─────────────────────────────────────────────────────────────┤
│  Current: 3M+ transactions/day                              │
│  Capacity: 100M+ transactions/day (theoretical)             │
│  Bottleneck: Gas prices rise with congestion                │
│  Solution: Layer 2 rollups when needed                      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  LAYER 2: Caching & Indexing                                │
├─────────────────────────────────────────────────────────────┤
│  Read operations: 99% served from cache (Redis)             │
│  Write operations: Batched when possible                    │
│  Proof generation: Client-side (scales with users)          │
│  Database: PostgreSQL + TimescaleDB (time-series)           │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  LAYER 3: Data Sources (External APIs)                      │
├─────────────────────────────────────────────────────────────┤
│  GitHub API: Rate limited to 5,000/hour (per token)         │
│  Solution: Multiple API keys, request pooling               │
│                                                             │
│  Oracle data: Cached for 24 hours (stale data acceptable)   │
│  Solution: Distributed oracle network                       │
└─────────────────────────────────────────────────────────────┘
```

**Result:** System scales horizontally, no central bottleneck.
